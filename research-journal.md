## Session 7 Journal

**Date:** May 2, 2026
**Project:** Text-to-Game Generator (HuggingFace Space)

---

### Overview

This session was a long and iterative journey building a fully working **AI-powered HTML5 game generator** hosted on HuggingFace Spaces. The app takes a theme and game genre, generates playable game code, and generates sprite images — all for free.

---

### What I Built

A HuggingFace Space (`LeafCat79/Game_Builder_Free`) that:
- Takes a **theme** (e.g. "Jungle temple with ancient traps") and **game genre** as input
- Uses an LLM to **generate complete, playable HTML5 canvas games**
- Uses an image AI pipeline to **generate themed sprites** (player, background, enemy)
- **Embeds sprites as base64** directly in the HTML so the game is fully self-contained
- Renders the game live in an **iframe** inside the Gradio UI

---

### Architecture (Final)

| Component | Model | Provider |
|---|---|---|
| Game code generation | `llama-3.1-8b-instant` | Groq API (free) |
| Image prompt enhancement | `llama-3.3-70b-versatile` acting as Z-Image-Engineer V4 | Groq API (free) |
| Sprite image generation | FLUX via Pollinations.AI | Pollinations (free, no key) |

---

### Problems Encountered & Solution

#### 1. Memory Limit Exceeded (OOM — 16GB)
**Problem:** Loading large models locally (Qwen3-Coder-Next ~140GB, ERNIE-Image ~32GB) crashed the Space container.
**Solution:** Switched to API-based inference — models run on external servers, Space uses ~200MB RAM.

#### 2. Build Timeout (Exit Code 143)
**Problem:** `llama-cpp-python` compiles from C++ source during Docker build, taking 20+ minutes and exceeding HF's build timeout limit.
**Solution:** Removed `llama-cpp-python` entirely, switched to API calls which need no compilation.

#### 3. HF Inference API Quota Exhausted (402 Error)
**Problem:** Free monthly HF credits depleted — every image call returned `402 Payment Required`.
**Solution:** Tried multiple providers:
- nscale → didn't host FLUX.1-schnell
- hf-inference → quota depleted
- Google Imagen-3 → needed Google Cloud project setup (unavailable)
- **Pollinations.AI** → completely free, no API key, no signup ✅

#### 4. Game Not Working (Black Screen / Player Falls)
**Problem:** Three bugs in AI-generated game code:
1. Black screen — images not loaded before game loop started
2. Player falls into void — `grounded=false` set inside `else` of platform loop
3. Keyboard not working — `keydown` listener on `canvas` instead of `window`

**Solution:** Added explicit CRITICAL RULES to the system prompt:
- Use `Promise.all()` to wait for all images before calling `gameLoop()`
- Set `grounded=false` **before** the platform loop, not inside `else`
- Add keyboard listeners on `window`, not `canvas`

#### 5. Sprites Showing as Colored Boxes with Text Labels
**Problem:** FLUX image generation was silently failing and falling back to SVG placeholders. The fallback was hiding the real error.
**Solution:** Added visible error reporting to the status message so the actual API error could be diagnosed. Discovered the real issue was the wrong provider (`nscale` doesn't host FLUX) and depleted HF quota.

#### 6. Wrong Provider for Models
**Problem:** Multiple models claimed to be on inference providers but weren't:
- `Qwen/Qwen3-Coder-Next` — local only, no inference provider
- `baidu/ERNIE-Image` — on `fal-ai` provider
- `Tongyi-MAI/Z-Image-Turbo` — ZeroGPU only
- `FLUX.1-schnell` on nscale — not supported

**Solution:** Verified each model's actual provider through the HF API and community reports before coding.

#### 7. ZeroGPU Requires HF PRO
**Problem:** ZeroGPU (free on-demand GPU) requires HuggingFace PRO subscription ($9/month).
**Solution:** Avoided ZeroGPU entirely by using external free APIs (Groq + Pollinations).

#### 8. Duplicate Score/Lives Display
**Problem:** Generated game code was rendering UI elements twice.
**Solution:** Added explicit instruction in system prompt to draw UI only once per frame.

### Key Lessons Learned

1. **Always verify inference provider availability** before writing code — many HF models have no hosted endpoint
2. **Free quotas are monthly** — HF inference credits run out fast with image generation
3. **No-key APIs exist** — Pollinations.AI proves you don't always need an API key
4. **LLM-generated game code needs strict rules** — physics, keyboard input, and image loading all require explicit instructions in the system prompt or the model makes common bugs
5. **Error visibility matters** — silent fallbacks hide real problems; always surface errors to the user

## Week 10 Journal

**Date:** May 16, 2026
**Project:** Text-to-Game Generator — `bobbyzhang133-star/Game_Builder_Free`

---

### Overview

Week 10 was the longest and most intensive development session of the project. The focus shifted from getting the pipeline working (Session 7) to making the generated games actually playable and polished. This involved fixing a long series of recurring code generation bugs, improving the sprite image pipeline, refining the UI, and writing up the research paper and documentation.

---

### 1. Game Type Reduction

Reduced the supported game genres from 5 down to **2 — Platformer and Top-Down Shooter only**. The other three genres (Puzzle/Maze, Arcade/Dodge, Surprise Me!) were removed because they had significantly higher bug rates and inconsistent outputs. Focusing on two well-defined genres allowed tighter system prompt control and more predictable results.

---

### 2. UI Changes

**Collapsible panels:** The generated code window and image prompts window were both converted to collapsible `gr.Accordion` panels that start closed. This saves screen space since users rarely need to see the raw HTML unless debugging.

**Fixed game window size:** The iframe displaying the game was fixed to exactly **800×450px** to match the HTML5 canvas size. Previously the iframe used `width:100%` which caused the game to appear with letterboxing or scrollbars.

**Game category dropdown:** Reduced to only Platformer and Top-Down Shooter.

---

### 3. Detailed Game Structure Rules Added to Prompts

**Top-Down Shooter rules added:**
- Monsters spawn at the top edge and fall straight down
- Monster size 32×32, speed 1–2 px/frame (slower than player)
- Player moves left/right only with WASD along the bottom
- Left mouse click fires a bullet straight upward at constant speed
- Bullet hits monster → both removed, score +10
- Monster reaches player → health -1, monster removed
- Spawn rate: one new enemy every 120 frames

**Platformer rules added:**
- All platforms drawn with `sprite_platform.png` via `ctx.drawImage`
- Goal sprite (`sprite_goal.png`) at the top-right platform — touching it = WIN screen
- 3 monsters patrolling back-and-forth on their own platforms, reversing at edges
- Monster collision → lives -1, player respawns at start
- Proper grounded/gravity/jump physics enforced

---

### 4. CODE_SYSTEM Prompt Evolution

The `CODE_SYSTEM` system prompt went through the most iterations of anything in this project. Each bug discovered in generated code led to a new explicit rule. Key rules added over Week 10:

| Rule Added | Bug it Fixed |
|---|---|
| `canvas` and `ctx` must be the VERY FIRST lines of script | Model declared them inside functions — variables invisible to each other |
| `player` declared at top level before `Promise.all` | Model declared player after `startGame()` ran — undefined on first frame |
| `Promise.all([loadImg(playerImg), loadImg(bgImg), loadImg(enemyImg)]).then(startGame)` | Black screen — images drawn before they loaded |
| `startGame()` only sets listeners, NEVER redeclares canvas/ctx | Model re-declared canvas inside startGame — separate local variable |
| `gameLoop()` defined at TOP LEVEL, not inside startGame | `requestAnimationFrame(gameLoop)` failed — gameLoop out of scope |
| `FOR TOP-DOWN SHOOTER: NO gravity, NO velY += 0.5` | Model applied platformer physics to top-down game — player fell off screen |
| Bullets use `vy: -10` only, NO `vx` component | Model computed diagonal bullet direction from mouse position |
| `ctx.drawImage(bgImg, 0, 0, canvas.width, canvas.height)` — always pass width/height | Background SVG tiled as 64×64 instead of filling the canvas |
| Enemies spawn every 120 FRAMES (frameCount variable), not milliseconds | Model used `setInterval(90)` = 90ms spawn rate = screen full of enemies |
| `onShoot` added in `startGame()`, `restartHandler` only added on game over | Both click handlers active simultaneously — click during gameplay triggered restart |
| `gameOver = true` then `return` to stop loop | Game over loop continued running requestAnimationFrame infinitely |
| Player starts at `canvas.width/2-24, canvas.height/2-24` | Model set `y: canvas.height - 24` — player spawned off screen at bottom edge |

---

### 5. Sprite Generation Pipeline Improvements

**Background removal via `rembg`:** Added `rembg[cpu]` post-processing to automatically remove backgrounds from all character and object sprites after FLUX generates them. Background scenes are excluded. This addressed the grey/black box visible around every character sprite.

**Stronger perspective prompts:** The seed descriptions fed to the Z-Image-Engineer were updated with game-type-specific perspective wording:

| Game Type | Character Prompt Style |
|---|---|
| Top-Down Shooter | `"top-down overhead game sprite, like Metal Slug or GTA 2 angle, head at top"` |
| Platformer | `"2D side-view platformer sprite, facing right, like Super Mario or Mega Man"` |

**Pollinations.AI rate limiting fix:** Added 20-second delays between sprite requests and retry logic with 120-second timeout to avoid `429 Too Many Requests` errors from Pollinations' anonymous tier (1 request per 15 seconds).

---

### 6. Observations Made During Testing

**Observation 1:** Sprites generated do not match the game type perspective. Top-down shooter sprites appear in front-facing standing poses instead of overhead bird's-eye view. Platformer sprites appear front-facing instead of side-view. This persisted across multiple prompt iterations with increasingly specific wording.

**Observation 2:** Tested with ChatGPT to confirm correct perspective is achievable with the right prompting. ChatGPT successfully generated a side-view platformer sprite and a top-down overhead sprite when given the same game type as context. This confirms the failure is specific to FLUX.1-schnell's training distribution, not the prompt design approach.

**Observation 3 (Space Marine theme test):** Input: Deep space asteroid mining station under alien parasite attack. Output: Such kind of bug barely happens anymore. UI generation is very unstable, and code did not go fully well. Background not filling canvas (tiling issue), player invisible, game over screen appeared immediately on start.

**Observation 4:** LLM-generated game code is very unstable. The same prompt produces different bugs across different generations. This is an open research question — how do we eliminate or reduce code generation bugs?

### Key Lessons from Week 10

1. **Prose descriptions are insufficient for executable code** — the model needs exact code patterns as templates, not descriptions of desired behavior
2. **Prompt chaining (LLM → LLM → image model) works but has limits** — the Z-Image-Engineer enhances prompts effectively for atmosphere and theme, but cannot override FLUX's training distribution for structural properties like perspective
3. **Silent fallbacks hide real problems** — making errors visible in the UI status was essential for diagnosing FLUX failures
4. **Game type reduction was the right call** — 2 well-constrained genres with tight system prompts outperform 5 loosely defined genres with generic prompts
5. **rembg[cpu] not rembg** — the `rembg` package alone fails without `onnxruntime`; must specify `rembg[cpu]` in requirements to install the correct backend
